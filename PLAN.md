# План разработки: градостроительный симулятор «Эпохи и сословия»

Рабочее название — замени на своё. Документ устроен так: сначала фиксируем концепцию и архитектурные решения, затем дизайн ключевых систем, затем поэтапный план с готовыми промтами для Claude Code. Каждая фаза заканчивается играбельным результатом — это важно, чтобы не строить полгода «фундамент» без игры.

---

## 1. Концепция

Премиум-игра для Steam (разовая покупка, без микротранзакций): изометрический градостроительный симулятор в духе Forge of Empires — город на сетке, прогрессия через эпохи от каменного века до будущего, производственная экономика «поставил → произвёл → собрал». В отличие от f2p-образца, пейсинг активный: короткие производственные циклы, плотная событийная петля, сессии по 30–90 минут без «приходи завтра». Ключевое отличие от FoE по механикам — **система сословий**, работающая на двух уровнях:

1. **Классы населения** (крестьяне → горожане → рабочие → …, названия меняются по эпохам). У каждого класса свои типы жилья, потребности в товарах и услугах, уровень счастья. Довольный класс даёт бонусы (производительность, налоги), недовольный — штрафы вплоть до забастовок.

2. **Фракции влияния** (знать/гильдии/духовенство, позже — корпорации, профсоюзы и т.д.). У каждой — шкала лояльности. Игрок принимает **эдикты** (политики), которые радуют одни фракции и злят другие. Низкая лояльность → негативные события, высокая → уникальные бонусы и здания.

Режим: сингл, релиз только в Steam через Tauri-обёртку. Веб-сборка остаётся как dev-среда (Vite dev server — самые быстрые итерации) и не публикуется. Архитектура симуляции всё равно изолируется — это дёшево и оставляет дверь к мультиплееру.

## 2. Технологический стек

| Слой | Технология | Почему |
|---|---|---|
| Язык | TypeScript (strict) | типобезопасность критична для симуляции |
| Сборка | Vite | быстрые итерации |
| Рендер карты | PixiJS v8 | WebGL, спрайты, тысячи объектов без просадок |
| UI поверх карты | React 18 | панели, окна, меню — DOM поверх canvas |
| Состояние UI | Zustand | лёгкий, без бойлерплейта |
| Симуляция | чистый TS-модуль без зависимостей | детерминизм, тестируемость, переносимость на сервер |
| Десктоп-обёртка | Tauri 2 | лёгкая (~10 МБ против сотен у Electron), Rust-ядро, доступ к ФС |
| Steam-интеграция | steamworks.js | ачивки, оверлей, Steam Cloud, Rich Presence |
| Сейвы | JSON-файлы через Tauri fs + Steam Auto-Cloud | синхронизация между устройствами бесплатно, без своего бэкенда |
| Аудио | Tone.js | генеративный lofi, раздел 8 |
| Тесты | Vitest | юнит-тесты симуляции обязательны |
| Контент | JSON/TS-конфиги (data-driven) | здания/эпохи/товары добавляются без кода |

В dev-режиме игра крутится в браузере (`npm run dev`) с сейвами в localStorage-заглушке; Tauri-специфика (файлы, Steamworks) — за интерфейсом `Platform`, чтобы dev-цикл оставался мгновенным.

Никаких готовых игровых движков (Phaser и т.п.) — для изометрии с таймерной экономикой связка Pixi + React даёт больше контроля и меньше лишнего.

## 3. Архитектура

Три жёстко разделённых слоя:

```
┌─────────────────────────────────────────────┐
│  UI (React): панели, окна, HUD, туториал    │
└──────────────▲──────────────────────────────┘
               │ selectors / commands
┌──────────────┴──────────────────────────────┐
│  SIMULATION (чистый TS): GameState,         │
│  детерминированный tick(), команды,         │
│  правила экономики, сословий, эпох          │
└──────────────▲──────────────────────────────┘
               │ snapshot / events
┌──────────────┴──────────────────────────────┐
│  RENDERER (PixiJS): изометрическая сцена,   │
│  спрайты, анимации, камера, ввод по карте   │
└─────────────────────────────────────────────┘
```

Принципы:

- **Симуляция ничего не знает о Pixi и React.** Она принимает команды (`PlaceBuilding`, `CollectProduction`, `EnactEdict`...) и выдаёт новое состояние + события.
- **Детерминированный тик.** 1 тик = 1 секунда игрового времени. Оффлайн-прогресс = «прокрутить N тиков при загрузке».
- **Команды валидируются внутри симуляции**, а не в UI.
- **Весь контент — данные.** Здание описывается JSON-объектом, код только интерпретирует.
- **Сейв = сериализованный GameState + версия схемы** и миграции между версиями.

### Структура проекта

```
src/
  sim/              # симуляция (без зависимостей)
    state.ts        # GameState, типы
    commands.ts     # команды и их валидация
    tick.ts         # главный цикл
    systems/        # economy.ts, population.ts, estates.ts,
                    # factions.ts, eras.ts, events.ts, quests.ts
    content/        # buildings.ts, eras.ts, goods.ts, edicts.ts
    save/           # serialize.ts, migrations.ts, offline.ts
  render/           # PixiJS
    scene.ts        # инициализация, слои
    iso.ts          # математика изометрии (grid<->screen)
    tilemap.ts      # земля, дороги
    buildings.ts    # спрайты зданий, анимации
    camera.ts       # pan/zoom, инерция
    input.ts        # клики/тапы по сетке, drag размещения
    atlas/          # генерация и загрузка атласов спрайтов
  ui/               # React
    hud/            # ресурсы, эпоха, кнопки
    panels/         # строительство, сословия, политика, древо
    windows/        # здание, событие, квест
  app/              # склейка: цикл, store, сейв-менеджер
assets/
  sprites-src/      # SVG-исходники зданий по эпохам
  atlases/          # собранные PNG-атласы
tools/
  build-atlas.ts    # SVG -> атласы (запускается отдельно)
```

## 4. Дизайн ключевых систем

### 4.1 Изометрическая карта

- Сетка ромбов, базовый тайл 64×32 px (классика FoE-стиля). Карта генерируется процедурно с рельефом (раздел 4.10), город растёт выкупом участков.
- Здания мультитайловые: 1×1 (декор), 2×2 (дома), 3×3 (производство), 4×4 (ратуша, чудеса). Поворот зданий не делаем (как в FoE) — экономит вдвое количество спрайтов.
- Z-сортировка спрайтов по `(row + col)` опорного тайла; для мультитайловых — по дальнему углу.
- Ввод: hover подсвечивает тайлы (зелёный/красный при размещении), drag — перемещение построенного здания, как в FoE.
- Камера: pan мышью/тачем, zoom колесом/щипком, плавная инерция, ограничение границами карты.

### 4.2 Дороги и связность

- Дороги — отдельный слой тайлов с **автотайлингом**: 16 вариантов спрайта по битовой маске соседей (N/E/S/W). Со средних эпох — дороги 2×2 для крупных зданий.
- Правило связности: здание работает, только если хотя бы один его тайл примыкает к дороге, соединённой с ратушей. Проверка — BFS по графу дорог от ратуши, пересчёт при изменении дорог (кэшируется).
- Несоединённые здания визуально маркируются (иконка разрыва) и не производят.

### 4.3 Экономика

- **Базовые ресурсы**: золото (налоги с жилья), припасы (производственные здания), население (жильё даёт, остальные здания потребляют рабочих).
- **Товары эпох**: у каждой эпохи 3–5 товаров. Производятся в мануфактурах из базовых ресурсов, тратятся на исследования, переход эпох и потребности сословий.
- Производство циклами 30 сек – 5 мин с выбором рецепта. Оффлайн-прогресс: КПД ~50%, экран «пока вас не было» при запуске.
- Правило Steam: возврат денег при < 2 часов игры. Первые два часа должны быть плотными.
- Цены и тайминги — в одном файле баланса `content/balance.ts`.

### 4.4 Сословия — классы населения

- В каждой эпохе 2–4 класса. У каждого класса свой тип жилых зданий.
- Потребности трёх уровней: базовые (еда, безопасность), комфорт (товары эпохи, рынки), роскошь (культурные здания, парки). Покрытие = % счастья класса.
- Эффекты счастья: > 80% — буст +20%, 50–80% — норма, < 50% — штрафы, < 25% — событие «волнения».
- Социальная мобильность: апгрейд жилья переводит жителей в следующий класс.

### 4.5 Сословия — фракции и политика

- 3 фракции на эпоху, шкала лояльности 0–100.
- **Эдикты** — активные политики (3 слота, кулдаун смены).
- Лояльность < 30 → негативные события. Лояльность > 80 → дары, уникальные здания фракции.
- **События** — карточки с выбором из 2–3 опций.

### 4.6 Эпохи и технологии

Полный список: каменный век → бронзовый → железный → античность → раннее средневековье → высокое средневековье → возрождение → колониальная → индустриальная → прогрессивная → модерн → постмодерн → информационная → будущее.

- **Древо технологий** внутри эпохи (12–20 узлов): открывает здания, эдикты, расширения.
- Переход эпохи: завершить ключевые технологии + построить «символ эпохи».
- Движок поддерживает все 14 эпох с первого дня; контент к релизу — 5–6 эпох.

### 4.7 Графика: детализированные здания без художника

1. **Процедурный SVG-пайплайн (старт).** Параметрические SVG-шаблоны зданий с палитрой эпохи. `tools/build-atlas.ts` рендерит SVG в PNG-атласы.
2. **Замена на ручную графику (потом).** Рендер берёт спрайты из атласов по id — художник заменяет PNG без кода.

Детализация: анимированный дым из труб, флаги, мерцающие окна, анимация «готово к сбору», жители-точки на дорогах.

### 4.8 Ресурсы: полная номенклатура

Семь базовых ресурсов + 42 товара.

| Ресурс | Источник | Расход | С какой эпохи |
|---|---|---|---|
| Золото | налоги с жилья | стройка, земля, дороги | с начала |
| Припасы | производственные здания | стройка, сырьё для товаров | с начала |
| Население | жильё | рабочие места зданий | с начала |
| Знания | библиотеки, университеты | древо технологий | с начала |
| Влияние | довольные сословия, лояльные фракции | эдикты, великие здания | с бронзового века |
| Энергия | электростанции | требование поздних зданий | с индустриальной |
| Товары | мануфактуры (3 вида на эпоху) | технологии, переход эпох, потребности сословий | по эпохам |

Товарные цепочки: каждый товар эпохи N использует в рецепте один товар эпохи N−1.

### 4.9 Великие здания

Цель — 2 на эпоху (28 всего). Три несущих решения:

- **Эффекты = данные.** Единая система модификаторов через `sim/systems/modifiers.ts`.
- **Разблокировка через фракции.** Чертёж даёт фракция при лояльности 80+.
- **Уровни 1–10.** Стоимость и эффекты в таблице `balance.ts`.

### 4.10 Процедурная карта и рельеф

Карта генерируется по seed. Размеры: 24×24 / 32×32 / 48×48.

Типы рельефа:
- **Река** — мосты через неё, мельницы/порты у воды.
- **Горы** — непроходимы, шахты у подножия.
- **Лес** — вырубить или сохранить (счастье жилью рядом).
- **Плодородная земля** — фермы производят больше.

### 4.11 Арка прохождения: вехи вместо финала

- **Эпоха = мини-финал.** Церемония перехода, экран «Итоги века», смена музыкальной темы.
- **Летопись** — автоматическая хроника событий по векам, хранится в сейве.
- **Вехи** — сетка достижений с наградами влияния, зеркалятся в Steam-ачивки.

**Режимы сложности:** мирный / нормальный / суровый.

**Бюджет производительности:** 60 FPS при 1000+ спрайтов, тик симуляции ≤ 2 мс.

## 5. Фазы разработки и промты для Claude Code

Правила работы: одна фаза = одна или несколько сессий Claude Code. После каждой фазы — ручная проверка в dev-режиме, затем коммит.

### Фаза 0 — каркас (полдня)

> Создай проект градостроительного симулятора по PLAN.md (разделы 2–3): Vite + TypeScript strict + React 18 + PixiJS v8 + Zustand + Vitest, поверх — Tauri 2 (минимальная конфигурация, окно 1280×800, иконка-заглушка). Создай структуру каталогов из раздела 3 с заглушками и типами ядра: GameState, Tile, Building, BuildingDef, Resources, Command. Добавь интерфейс Platform (saveFile/loadFile/listSaves) с двумя реализациями: localStorage для dev-браузера и Tauri fs для сборки. Настрой ESLint/Prettier. Сделай страницу, где Pixi-canvas занимает весь экран, а React рендерит поверх пустой HUD. Напиши первый тест: tick() с пустым состоянием не падает и инкрементирует время. Критерий готовности: npm run dev показывает сцену в браузере, npm run tauri dev — то же в нативном окне, npm test зелёный.

### Фаза 1 — изометрический движок (2–4 сессии)

> Реализуй изометрическую карту по PLAN.md 4.1. (1) Математика iso.ts: gridToScreen/screenToGrid для тайла 64×32, юнит-тесты на оба направления. (2) Процедурная генерация карты по PLAN.md 4.10: seed-генератор (трава, река, горы, лес, плодородные земли), стартовая зона 8×8 гарантированно чистая; tilemap с рендером чанками и автотайлингом краёв рельефа; тесты генератора (валидность, детерминизм по seed). (3) Камера: pan перетаскиванием, zoom 0.5–2 к курсору, инерция, границы. (4) Размещение зданий: временные процедурные спрайты-боксы 1×1, 2×2, 3×3 с крышей; режим строительства из панели — призрак под курсором, подсветка валидности, клик ставит; занятость тайлов в GameState через команду PlaceBuilding с валидацией в симуляции. (5) Z-сортировка мультитайловых зданий — тест на перекрытия. (6) Перемещение построенного здания перетаскиванием. Критерий: могу поставить 20 зданий разных размеров, двигать их, перекрытия корректны на любом зуме; стресс-сцена из 1000 зданий держит 60 FPS, тик ≤ 2 мс.

### Фаза 2 — дороги и связность (1–2 сессии)

> Реализуй дороги по PLAN.md 4.2: слой дорог, автотайлинг по 4-битной маске соседей (сгенерируй 16 процедурных спрайтов), инструмент рисования дорог протягиванием, удаление. Затем связность: BFS от ратуши (добавь здание ратуши 4×4, ставится автоматически в центре при новой игре), флаг connected у зданий, иконка разрыва над несоединёнными, пересчёт при любом изменении дорог с кэшем. Тесты на BFS и автотайлинг.

### Фаза 3 — экономика и сейвы (2 сессии)

> Реализуй экономическое ядро по PLAN.md 4.3: ресурсы (золото, припасы, население), жилые здания дают население и налоги по циклу, производственные требуют рабочих и производят припасы с выбором рецепта (30 сек – 5 мин). Окно здания (React) с прогрессом и кнопкой сбора, всплывающая иконка сбора над зданием. Контент: 8 зданий каменного века в content/buildings.ts. Сейвы: автосейв через интерфейс Platform каждые 30 сек и при закрытии (3 слота + автослот), загрузка при старте, оффлайн-прогресс с КПД 50%, экран «пока вас не было». Версия схемы сейва + каркас миграций. Тесты: производственный цикл, оффлайн-прогресс, сериализация туда-обратно.

### Фаза 4 — классы населения (2 сессии)

> Реализуй классы населения по PLAN.md 4.4 для двух эпох: численность по типам жилья, потребности трёх уровней, расчёт счастья, эффекты (буст/штраф/волнения как событие-карточка). Панель «Сословия» в HUD: доли классов, счастье, разбор непокрытых потребностей с подсказками что строить. Социальная мобильность через апгрейд жилья. Все коэффициенты — в content/balance.ts. Тесты на расчёт счастья и пороги эффектов.

### Фаза 5 — фракции, эдикты, события (2 сессии)

> Реализуй фракции по PLAN.md 4.5: 3 фракции, лояльность, 9 эдиктов (3 слота, кулдаун смены), панель «Политика». Система событий: пул карточек с условиями появления, окно выбора с последствиями, генерация негативных событий при низкой лояльности и дары при высокой. Контент: 15 событий для первых эпох. Тесты на триггеры событий и эффекты эдиктов.

### Фаза 6 — эпохи и древо технологий (2–3 сессии)

> Реализуй эпохи по PLAN.md 4.6: очки знаний, древо технологий с UI (зум-панель, ветки, статусы), разблокировка зданий/эдиктов, переход эпохи с условиями и церемонией (символ эпохи, смена названий классов и пула фракций). Контент: полные древа и здания для каменного, бронзового и железного веков; каркас остальных эпох списком. Тесты на разблокировки и условия перехода.

### Фаза 6.5 — великие здания (1–2 сессии)

## 8. Аудио: генеративный lofi-движок на Tone.js

Музыка генерируется в рантайме на Web Audio — бесконечный плейлист, ноль мегабайт ассетов, звучание привязано к состоянию игры.

- **Гармония**: банк lofi-прогрессий (ii–V–I, I–vi–IV–V, vi–IV–I–V и джазовые варианты), смена прогрессии раз в 4–8 тактов.
- **Инструменты**: электропиано (FM-синтез под Rhodes), мягкий синус-бас, пентатоническая мелодия, свинговые ударные.
- **Lofi-фактура**: виниловый шум, LPF на мастере (~3–5 кГц), vibrato LFO, псевдо-сайдчейн.
- **Связь с игрой**: каждая эпоха задаёт пресет — лад, темп, набор инструментов. Цикл дня/ночи замедляет темп ночью на ~8 BPM. События «волнений» приглушают музыку.
- **Технические правила**: AudioContext стартует только после первого жеста; `src/audio/` изолирован, симуляция о нём не знает.
